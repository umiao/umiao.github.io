SQL Study Note - 1 - Syntax Basics

2022-04-16
2022-04-16
DataScience, SQL

Overview

  1. The core of database system is to interact with DB (DataBase) with DBMS (Database Management System).

  1. The DB can be generally divided into:
    • Relational DB
    • NoSQL DB (e.g., KV (key-value) based DB)

  1. Recommended code style of SQL (Structed Query Language):
    • Capitalize all keywords and reserved words, and lowercase all other contents.
    • Each statement should end with ;.

Keywords and Syntax Rules

The query syntax

  1. SHOW DATABASES List all the names of exisitng databases (under the current schema).

  1. USE Select one DB as the default one.

  1. SELECT Select the desired data from a table.
    It should be noted that you should use = in SQL to determine equivalence. (It DOES NOT mean assignment.)
    You can conduct calculation on the selected result.

  1. AS For each col_name, table_name to be queried and the queried results, you can always to set alias (surname) for them with AS.

  1. – / comment

  1. DISTINCT Add DISTINCT ahead of the queried target (column) to receive all the unique values.

  1. AND / OR / NOT Logical operator to be used with the WHERE clause.

  1. IN Determine if the queried value belongs to a set.

  1. BETWEEN Determine if the queried value within a given interval.
    Both ends of the interval are closed ([beg, end]).

  1. LIKE Provide functionality similar to Regular Expression. % can match arbitrary string, _ can match arbitrary char. Not sensitive to the case.

  1. REGEXP Match a given Regular Expression pattern.

  1. IS NULL Query all the records which are (not) null in a given column.

  1. ORDER BY Decide the sorting order of the returned records.

  1. LIMIT Restrict the number of the returned records.
    Skip the first $n=$offset records and then return tot_num lines of records.
    Return all the records if number of matched records fewer than tot_num.

  1. INNER JOIN Can be simply writtene as JOIN. Concat two tables based on the given condition. Conduct Cartesian product implicitly.

  1. SELF JOIN A table can join with itself, but different alias are required.

  1. Multi-table JOIN Multiple (n) tables can be joined but this is not recommended when $ n > 3 $ due to performance concern.

  1. Composite JOIN / Implicit JOIN Sometimes only a tuple of multiple attributes can uniquely identify a row of the table. In this case, these attributes become Composite Primary Key. Implicit JOIN is conducted in the above example but this is not recommended, too.

  1. OUTER JOIN When using INNER JOIN, some records of the left table cannot match with the right table because the condition of the ON clause is not satisfied. However, if we want to return all the records of the left (right) table regardless of the boolean value of the ON clause, we can use LEFT (RIGHT) OUTER JOIN.
    You should use RIGHT JOIN instead of LEFT JOIN as possible.
    SELF OUTER JOIN is similar, and alias is still required.

  1. USING Can be used to simplify the code, if the column names of the to-be-joined tables are exactly the same. You can Join on a tuple like ‘USING (id1, id2, id3)’ and these column names should be exactly the same as well (in the two tables).

  1. NATURAL JOIN Let the compiler (DBMS) to decide the way of join. Not recommended to use!

  1. CROSS JOIN Conduct Cartesian Product.

  1. UNION Concatenate multiple queried results together (on the direction of row). The column names should be exactly the same.
    IT SHOULD BE NOTED that ORDER BY can be set only once, so union all the results before setting the ORDER clause.
